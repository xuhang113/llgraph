# 大模型协议对接与模型适配

本文说明 llgraph 与 **AI Gateway**（Anthropic Messages 兼容形态）对接时的消息协议、`additional_kwargs` 的含义，以及各模型的出站特殊规则。

相关文档：[会话上下文与历史.md](会话上下文与历史.md)、[模块说明.md](模块说明.md)。

---

## 1. 整体架构：三层消息形态

llgraph 同一段对话在链路中会经历三种形态，职责不同：

```text
┌─────────────────────────────────────────────────────────────────┐
│ ① LangChain 内存 / messages.jsonl（canonical v2 落盘）           │
│    Human / AI / Tool + additional_kwargs.llgraph                 │
├─────────────────────────────────────────────────────────────────┤
│ ② 出站修链后（MessageDispatchProfile，按当前 /model）            │
│    展开 tool 链、补 reasoning、剥 thinking                          │
├─────────────────────────────────────────────────────────────────┤
│ ③ HTTP Request Body（ChatAnthropic._get_request_payload）        │
│    system + messages[]；Kimi 另需顶层 reasoning_content 注入     │
└─────────────────────────────────────────────────────────────────┘
```

**原则**：落盘用一种 **canonical** 格式；**每轮发 API 前**再按当前模型做 adapter。避免「按 Kimi 落盘、按 Claude 落盘」来回改磁盘。

## 2. `additional_kwargs` 是什么

### 2.1 LangChain 层面的定义

`additional_kwargs` 是 LangChain `BaseMessage`（含 `AIMessage`）上的 **扩展字段字典**，用于承载：

- 厂商 API 返回、但尚未被 LangChain 升格为一等公民的字段；
- SDK / 网关特有的 metadata（如 `usage`、流式 delta 残留等）；
- 框架或业务自行挂载的侧车信息。

它 **会参与** `messages_to_dict` / `messages_from_dict` 序列化，因此会写入 `messages.jsonl`，但 **不保证** LangChain 在构造 HTTP 请求时会把其中每个键都映射到 API body。

### 2.2 llgraph 在 `additional_kwargs` 里存什么

| 键 | 写入时机 | 用途 | 是否上送网关 |
|----|----------|------|--------------|
| `llgraph.thinking_text` | 落盘 canonical 时，从 AI 的 thinking / reasoning 块剥离 | 会话检索、压缩锚点、Kimi 出站时回填 reasoning | 否（经 adapter 转为顶层 `reasoning_content` 或丢弃） |
| `llgraph.*` 其它 | 会话元数据（如 trace 摘要） | 落盘 / 检索 | 否 |
| `reasoning_content` | **仅出站** dispatch 时临时写入 | Kimi thinking 模式校验 | **需** `gateway_kimi_patch` 注入到 HTTP `messages[].reasoning_content` |

**落盘规范**（`message_canonical.py`）：

- AI 的 `content` 扁平为 **字符串**（去掉 thinking / tool_use 等块）；
- 思考过程写入 `additional_kwargs.llgraph.thinking_text`；
- **落盘时移除** `additional_kwargs.reasoning_content`（避免按某一网关污染持久化形态）。

### 2.3 为什么光有 `additional_kwargs` 不够

LangChain 的 `ChatAnthropic._format_messages()` 会把消息转成 Anthropic 风格的 `role` + `content` 数组，**默认不会**把 `additional_kwargs.reasoning_content` 写到 HTTP body 顶层。

Kimi k2.x 网关在 **thinking 开启** 时要求：历史里每条 **带 tool 的 assistant**，请求体中必须有 **`reasoning_content` 字段**（与 `content` / `tool_use` 同级）。

因此 llgraph 有两层处理：

1. **对象层**：`chat_history_repair.patch_tool_ai_reasoning_contents` → 写入 `AIMessage.additional_kwargs.reasoning_content`；
2. **HTTP 层**：`gateway_kimi_patch.inject_reasoning_into_formatted_messages` → 写入 `payload["messages"][i]["reasoning_content"]`。

仅做第 1 层仍会 400；第 2 层必须按 **formatted 里含 `tool_use` 的 assistant** 扫描注入（不能简单按 AIMessage 条数对齐，LangChain 可能跳过空 assistant 导致槽位错位）。

## 3. 基本协议：OpenAI Chat vs Anthropic Messages

llgraph 统一使用 **`ChatAnthropic` + 公司 Gateway `base_url`**，底层按 **Anthropic Messages** 形态组包；Gateway 再转发到各厂商。理解两套「公约数」有助于排查 400。

### 3.1 消息角色与顺序

| 角色 | 典型用途 | 顺序约束（严格网关常见） |
|------|----------|---------------------------|
| `system` | 系统提示、manifest、anchor | Anthropic：独立 `system` 字段或首条 system；llgraph 合并为 **一条** `SystemMessage` |
| `user` | 用户输入；Anthropic 下 **tool_result** 也放在 user 消息的 content 块里 | 一般 flexible |
| `assistant` | 模型正文 + `tool_use` 块 | 带 tool 时 content 非空（或仅最后一条 assistant 可为空） |
| `tool`（LangChain `ToolMessage`） | 工具执行结果 | OpenAI：可「1 AI + 连续多 Tool」；**火山 / Kimi 等**：每条 Tool 前一条 **仅含对应 call 的 AI** |

### 3.2 Tool Calling 两种常见形态

**OpenAI Chat Completions 风格（宽松）**

```text
user → assistant(tool_calls: [A, B]) → tool(A) → tool(B) → user → …
```

**Anthropic / 严格网关风格（llgraph 称 expand_parallel_tool_rounds）**

```text
user → assistant(tool_call: A) → tool(A) → assistant(tool_call: B) → tool(B) → user → …
```

llgraph 在 dispatch 阶段把「1 个 AI + N 个 Tool」展开为 N 组「AI(单 call) + Tool」，见 `chat_history_repair._append_single_tool_round`。

### 3.3 Thinking / Reasoning 的表示差异

| 厂商/系列 | 思考内容常见载体 | 历史 tool 轮是否强制带 reasoning |
|-----------|------------------|----------------------------------|
| **Kimi k2.x** | 流式 thinking；网关校验 **`reasoning_content` 顶层字段** | **是**（thinking 开启时） |
| **Anthropic Claude** | content 内 `thinking` / `redacted_thinking` 块 | 否（块结构不同） |
| **OpenAI / GPT 系** | o 系列 / 部分网关的 reasoning 字段 | 多数不强制历史补齐 |
| **GLM / DeepSeek / Minimax 等** | 多为 content 内 thinking 块 | 通常不强制 `reasoning_content` |

### 3.4 llgraph 出站管线（每轮 API 前）

```text
prepare_messages_for_llm_dispatch()
  → to_canonical_v2_messages()
  → sanitize_chat_history_for_dispatch(profile)   # 按当前模型
  → trim_messages_for_dispatch_window()
  → normalize_messages_for_llm()                  # 合并 system
  → ChatAnthropic._get_request_payload()
  → [Kimi] gateway_kimi_patch 注入 reasoning_content
```

Profile 解析顺序（`message_dispatch_profile.py`）：

1. `agent.json` → `llm.models[].dispatch`
2. `llm.message_dispatch_defaults`
3. 模型 id 启发式（`kimi-k2` / `glm` / `claude` / `gpt` 等）

`/model <id>` 会 `rebuild_agent_preserving_memory()`：**保留 messages**，只换 LLM 实例与 outbound profile。

## 4. MessageDispatchProfile 字段说明

配置在 `.llgraph/agent.json` → `llm.models[].dispatch`（或 `message_dispatch_defaults`）。

| 字段 | 含义 | 典型适用模型 |
|------|------|--------------|
| `expand_parallel_tools` | 1 AI + 多 Tool → 多组 AI(单 tool) + Tool | Kimi、GLM、DeepSeek、Minimax |
| `patch_reasoning_content` | 为带 tool_calls 的 AI 补 `reasoning_content`（对象层 + 触发 HTTP 注入） | Kimi k2.5 / k2.6 |
| `strip_assistant_thinking_blocks` | 去掉 content 内 thinking / tool_use 等块，收成纯文本 | GPT、Claude |

**落盘**始终用 `canonical_persist_profile()`（不展开、不补 reasoning），与出站分离。

### 4.1 工作区默认配置示例

```json
{
  "llm": {
    "model": "kimi-k2.6",
    "message_dispatch_defaults": {
      "expand_parallel_tools": false,
      "patch_reasoning_content": false
    },
    "models": [
      {
        "id": "kimi-k2.6",
        "dispatch": {
          "expand_parallel_tools": true,
          "patch_reasoning_content": true
        }
      },
      {
        "id": "claude-sonnet-4-6",
        "dispatch": {
          "expand_parallel_tools": false,
          "patch_reasoning_content": false,
          "strip_assistant_thinking_blocks": true
        }
      }
    ]
  }
}
```

## 5. 各模型特殊规则速查

### 5.1 Kimi k2.5 / k2.6

| 项 | 规则 |
|----|------|
| thinking | 网关侧 k2.6 **默认开启**；多轮 tool 需 `payload.thinking = {type: enabled, keep: all}` |
| reasoning 回传 | 历史 **带 tool 的 assistant** 必须同时有：① 顶层 `reasoning_content`；② `content` 内 `thinking` 块（Anthropic 形态仅写 ① 仍会 400） |
| tool 链 | 建议 `expand_parallel_tools: true` |
| llgraph 补丁 | `gateway_kimi_patch.py`：注入 `reasoning_content` + `content.thinking` + `thinking.keep=all` |
| 思考来源 | 优先 API 原始 thinking 块 → `llgraph.thinking_text` → 可见正文 → 占位符 |
| 常见 400 | `reasoning_content is missing at index 2`：多为缺 `thinking` 块或未设 `keep=all`，而非未重启 |
| 兜底 | `thinking: {type: disabled}` 可绕过校验，但会关闭模型思考输出 |

### 5.2 Claude（Opus / Sonnet 4.6）

| 项 | 规则 |
|----|------|
| thinking 块 | 原生 API 用 content 内块；**回传历史**时 llgraph **剥离** thinking 块 |
| tool 历史 | **保留** LangChain `AIMessage(tool_calls)` + `ToolMessage` 链；出站时将含 `:` 等的 `tool_call_id` 规范为 `grep_files_65` 等形式 |
| expand | **false**：并行 tool 时「1 assistant + 多 tool_use → 1 user + 多 tool_result」，符合 [Parallel tool use](https://platform.claude.com/docs/en/agents-and-tools/tool-use/parallel-tool-use) |
| reasoning_content | **不补** |

### 5.3 GPT-5.4 等 OpenAI 系

| 项 | 规则 |
|----|------|
| thinking 块 | 出站剥离（同 Claude 的 strip 策略） |
| tool 链 | 通常不展开 |
| reasoning_content | **不补** |

### 5.4 GLM / DeepSeek / Minimax

| 项 | 规则 |
|----|------|
| tool 链 | `expand_parallel_tools: true`（严格「Tool 前必有 AI」） |
| reasoning_content | **不补** |
| thinking | 可能在 content 内；落盘时收到 `thinking_text`，出站一般保留块或文本视网关而定 |

## 6. HTTP 层：Kimi `reasoning_content` 注入

实现：`llgraph/gateway_kimi_patch.py`（在 `create_gateway_llm()` 时 patch `ChatAnthropic._get_request_payload`）。

**正确做法**：

- 遍历 `payload["messages"]`；
- 对 `role == "assistant"` 且 content 中含 `type: "tool_use"` 的块；
- 若缺少非空 `reasoning_content`，从对应 source `AIMessage` 解析并写入。

**错误做法（已修复）**：

- 按 source 中 **每条** `AIMessage` 与 formatted 中 **每条** assistant 递增对齐；
- LangChain `_format_messages` 会 **skip** 中间空 assistant → 槽位错位 → 带 tool 的 assistant 漏注入。

兜底：注入后仍缺失则写占位符  
`（历史思考过程未落盘，占位以满足 Kimi 等网关校验。）`

## 7. 同会话切换模型

历史是 **canonical v2 落盘**（中性形态）。切换 `/model` 后：

- **不会**改写磁盘上的 messages.jsonl 为「Claude 形」或「Kimi 形」；
- **会**在下一轮 `prepare_messages_for_llm_dispatch` 时按 **新模型 profile** 临时修链。

示例：

- Kimi 上产生的 tool 历史 → 切到 Claude：出站时 strip thinking + 剥离 Kimi `reasoning_content`；
- Claude 上产生的历史 → 切回 Kimi：出站时 expand + 补 reasoning + HTTP 注入。

因此 **同一 thread_id 内切换模型**是支持的设计，依赖 outbound adapter 而非落盘多份。

## 9. 排查清单

### 9.1 Kimi 400：`reasoning_content is missing`

1. 是否 `pip install -e .` 且 **重启 llgraph 进程**（patch 在 import 时注册）；
2. 当前模型是否为 k2.5/k2.6，`patch_reasoning_content: true`；
3. 运行  
   `python scripts/check_model_dispatch_compat.py --probe-roundtrip`  
   确认 payload 层无缺失；
4. 长会话：历史 AI 是否仅有 content 字符串、无 `tool_calls` 但有 Tool 跟随（修链应补全或删孤儿 Tool）。

### 9.2 Tool 链 400

1. 试开 `expand_parallel_tools: true`；
2. 看 `[llgraph] 历史消息已修复` 日志是否含「展开并行工具轮」。

### 9.3 Claude / GPT 400（thinking、tool 相关）

1. 确认 `strip_assistant_thinking_blocks` 已开（跨模型 history 含 Kimi thinking 块时）；
2. 是否把 Kimi 形态的历史（含 reasoning 侧车）未经 dispatch 直发。
3. 并行 tool 勿开 `expand_parallel_tools`（会破坏 Anthropic 单 user 多 tool_result 形态）。

## 10. 术语对照

| 术语 | 含义 |
|------|------|
| **canonical v2** | llgraph 落盘/加载用的统一消息形态 |
| **dispatch / outbound** | 发往 Gateway 前临时修链，不写回 disk |
| **additional_kwargs** | LangChain 消息扩展字典；llgraph 主要用 `llgraph.*` 与临时 `reasoning_content` |
| **reasoning_content** | Kimi 网关要求的 assistant 顶层思考字段（HTTP body） |
| **thinking_text** | llgraph 落盘的思考文本（`additional_kwargs.llgraph`） |
| **expand_parallel_tools** | 并行多 tool 拆成多轮 AI+Tool |
| **Prompt Cache** | Anthropic 兼容的 `cache_control` 断点，见 [会话上下文与历史.md §10](会话上下文与历史.md) |
