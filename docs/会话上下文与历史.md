# 会话上下文与历史

本文说明 llgraph 的**历史会话存储格式**、**加载与出站流程**、**Token 节约机制**与**上下文压缩（Tier1～3）**。实现代码以 `llgraph/` 包为准；配置见工作区或用户目录下的 `.llgraph/agent.json` → `context` 段。

相关文档：[模块说明.md](模块说明.md)、[cursor-agent.md](cursor-agent.md)、[操作手册.md](操作手册.md)、[大模型协议对接与模型适配.md](大模型协议对接与模型适配.md)。

---

## 1. 设计原则（对齐 Cursor 思路）

| 层 | 做什么 | 是否消耗 API Token |
|----|--------|-------------------|
| **磁盘落盘** | 完整对话链 + 侧车 JSON/归档 | 否 |
| **进程内存** | LangGraph `MemorySaver`（按 `thread_id`） | 否 |
| **每轮发往模型** | 裁剪 / 压缩 / 出站窗口 / 按模型修链 | **是（主要成本）** |

核心约定：

1. **落盘形态统一**（canonical v2），不按某一网关写死。
2. **出站前再适配**（`MessageDispatchProfile`），兼容 Kimi / Claude / GPT 等。
3. **远早对话不进每轮上下文**：靠置顶摘要 + `search_session_history` + `read_file` 归档按需拉取。

## 2. 存储目录结构

会话数据在**用户目录**，按工作区隔离（不污染业务仓库）：

```text
~/.llgraph/context/<工作区slug>/
├── sessions/
│   ├── <thread_id>/
│   │   ├── messages.jsonl          # 当前会话正文（canonical v2）
│   │   ├── manifest.json           # Skills/Rules 目录 + 指针
│   │   ├── conversation_anchor.json # 结构化压缩锚点
│   │   ├── meta.json               # 标题、条数、格式版本等
│   │   ├── edits.jsonl             # 写操作账本（可选）
│   │   └── snapshots/              # 首次编辑前快照（可选）
│   └── <thread_id>.jsonl           # 压缩时导出的完整对话归档（与 thread 目录同级）
└── workspace.json                  # 工作区路径标记

<工作区>/.llgraph/context/tool-results/   # 超大 tool 结果落盘（相对工作区）
```

- **工作区 slug**：`{目录名}-{工作区hash前8位}`，例如 `my-monorepo-a1b2c3d4`。
- **thread_id**：交互模式默认 `cli-{8位hex}`；`llgraph --thread-id <id>` 可指定恢复。

旧版曾落在 `<工作区>/.llgraph/sessions/`，启动时会尝试迁移到用户目录（见 `user_storage.migrate_legacy_workspace_session_dir`）。

## 3. 历史会话格式

### 3.1 messages.jsonl（主存储）

- **一行一条消息**，LangChain `messages_to_dict` / `messages_from_dict` 序列化。
- 加载 / 落盘时统一经过 **`to_canonical_v2_messages()`**（`message_canonical.py`）。
- `meta.json` 中 `messages_format` 为 `canonical_v2`。

**canonical v2 约定：**

| 类型 | 落盘规则 |
|------|----------|
| `HumanMessage` / `ToolMessage` | 原样保留（tool 链完整） |
| `AIMessage` | `content` 扁平为字符串；thinking / reasoning 写入 `additional_kwargs.llgraph.thinking_text` |
| `SystemMessage`（业务中段） | 合并进一条 `<session-system-archive>`（若有） |
| 置顶 `SystemMessage` | 仅保留最新一条 `<session-manifest>`、一条 `<conversation-anchor>` |

典型置顶 manifest 片段：

```text
<session-manifest>
会话: cli-xxxxxxxx
manifest: `.../manifest.json`
对话归档: `.../cli-xxxxxxxx.jsonl`（可用 search_session_history / read_file）
结构化锚点: `.../conversation_anchor.json`
历史检索工具: search_session_history（按 query 搜 archive/messages/anchor 章节）
...
</session-manifest>
```

**注意**：`build_system_prompt()` 生成的 Agent 系统提示**不写入** jsonl，仅在每轮请求时由 `create_react_agent(prompt=...)` 注入。

### 3.2 manifest.json（侧车）

Skills / Rules 的**目录与路径句柄**（非正文），以及：

| 字段 | 含义 |
|------|------|
| `archive_path` | 最近一次压缩导出的归档 jsonl（相对或绝对路径） |
| `anchor_path` | `conversation_anchor.json` |
| `spill_dir` | 工具大结果目录（默认 `.llgraph/context/tool-results`） |
| `skills` / `rules` | 本回合目录项 |

每轮用户消息前会 `sync_session_manifest_to_agent_state()`，刷新置顶 `<session-manifest>`。

### 3.3 conversation_anchor.json（结构化锚点）

压缩后由 LLM 增量维护的章节（`conversation_anchor.py`）：

| 章节键 | 含义 |
|--------|------|
| `session_goal` | 会话目标 |
| `files_modified` | 已改文件列表 |
| `decisions` | 关键决策与结论 |
| `errors_resolved` | 错误与处理 |
| `pending_tasks` | 未完成与下一步 |
| `related_code` | Tier3 代码检索要点 |
| `detail_pointers` | archive / spill / 检索工具指引 |

对话中对应置顶 `<conversation-anchor>` SystemMessage。

### 3.4 压缩归档 `sessions/<thread_id>.jsonl`

压缩时导出的**压缩前全量**快照（`context_compressor._export_session_archive`），每行简化 JSON：

```json
{"role": "user|assistant|tool|system", "content": "...", "tool_calls": ...}
```

用于 `read_file` 全文回放或 `search_session_history` 关键词检索。

## 4. 加载与恢复流程

### 4.1 进程内记忆（MemorySaver）

```text
build_agent(with_memory=True)
  → create_checkpointer() → 单例 MemorySaver（无 SQLite）
  → create_react_agent(..., checkpointer=MemorySaver)
```

- 同一进程内，**同一 `thread_id`** 的多轮对话追加在 checkpointer 状态中。
- **进程退出后内存丢失**；跨重启依赖 jsonl 恢复。

### 4.2 启动 / 切换会话

```text
llgraph -C <工作区> --thread-id <id>
  或交互中 /session use <id> / /session new
        │
        ▼
session_is_resumable() 检查 messages.jsonl 是否存在
        │
        ▼
restore_session_to_agent(agent, workspace, thread_id)
  1. 读取 messages.jsonl
  2. messages_from_dict → to_canonical_v2_messages（必要时回写规范化）
  3. agent.update_state({ messages })
        │
        ▼
sync_session_manifest_to_agent_state()  # 置顶 manifest
```

`/session use` **不重建 Agent**，只换 `thread_id`、spill、edit_tracker 并恢复 jsonl。

### 4.3 每轮对话（invoke_agent）

```text
用户输入
  │
  ├─ [有 memory] ensure_session_title_auto
  │
  ├─ apply_incremental_tool_prune_to_agent_state   # 旧 Tool 变指针
  │
  ├─ apply_compress_to_agent_state (auto)          # 超阈值则 Tier1～3 压缩
  │
  ├─ sync_session_manifest_to_agent_state
  │
  ├─ ensure_agent_chat_history_sanitized (canonical 落盘清理)
  │
  ├─ set_active_thread_id(thread_id)               # 供 search_session_history
  │
  ├─ stream_agent_turn → LangGraph ReAct 循环
  │     └─ prompt normalizer:
  │           prepare_messages_for_llm_dispatch()
  │             → canonical v2
  │             → sanitize_chat_history_for_dispatch (按模型 profile)
  │             → trim_messages_for_dispatch_window (出站窗口)
  │             → 合并 agent system + manifest + anchor → 单条 SystemMessage
  │
  └─ persist_agent_session → save_session_messages (canonical v2)
```

## 5. 发往模型前的出站管线

与落盘分离，避免「按 Claude 形态落盘、按 Kimi 形态落盘」来回折腾。

```text
MemorySaver / jsonl 中的 messages
        │
        ▼
to_canonical_v2_messages()                    # 落盘/加载亦走此步
        │
        ▼
sanitize_chat_history_for_dispatch()          # 按 MessageDispatchProfile
  · expand_parallel_tool_rounds (Kimi 等)
  · patch_tool_ai_reasoning (Kimi reasoning_content)
  · strip_assistant_thinking_blocks (GPT/Claude)
  · rewrite_tool_history_as_text (Claude 历史 tool→Human 摘要)
        │
        ▼
trim_messages_for_dispatch_window()           # dispatch_keep_user_turns
        │
        ▼
normalize_messages_for_llm()                  # 合并为一条 system + 业务消息
        │
        ▼
ChatAnthropic / Gateway
```

**Profile 解析顺序**（`message_dispatch_profile.py`）：

1. `agent.json` → `llm.models[].dispatch`
2. `llm.message_dispatch_defaults`
3. 模型 id 启发式（kimi / glm / claude / gpt 等）

`/model <id>` 切换后会 `rebuild_agent_preserving_memory()`，保留 messages，仅换 LLM 与 normalizer 使用的 profile。

## 6. Token 节约方式

### 6.1 分层对照

| 手段 | 时机 | 作用 | 配置项（agent.json → context） |
|------|------|------|-------------------------------|
| **出站窗口裁剪** | 每轮 API 前 | 只发置顶 + 最近 N 个 user 轮 | `dispatch_keep_user_turns`（0=关闭） |
| **增量 Tool 裁剪** | 每轮 invoke 前/后 | 旧 ToolMessage 替换为「已落盘，read_file」指针 | `incremental_tool_prune`, `keep_recent_tool_messages`, `compress_tool_mask_max_chars` |
| **工具结果 Spill** | 工具返回时 | 超长结果写磁盘，上下文只留预览 | `spill_enabled`, `tool_result_max_chars`, `spill_dir` |
| **自动压缩** | invoke 前超阈值 | 删掉远早轮次，生成 anchor + 归档 | `auto_compress_ratio`, `compress_trigger_max_tokens`, `keep_recent_turns`, `keep_recent_token_ratio` |
| **手动压缩** | `/compress` | 强制走压缩器 | 同上 |
| **历史检索工具** | 模型按需调用 | 不全量重放；按 query 搜 archive/messages/anchor | `session_history_search_enabled`, `session_history_search_top_k` |
| **Claude 出站** | dispatch | 历史 tool 轮改文本摘要（不落盘改形） | `llm.models[].dispatch.rewrite_tool_history_as_text` |

### 6.2 search_session_history

- **工具名**：`search_session_history`
- **何时用**（已写入 system prompt / manifest）：用户指代「之前/刚才」、压缩或换模型后背景不清、anchor 不够细。
- **数据源**：`conversation_anchor.json` 各章节、`sessions/<id>.jsonl` 归档、`messages.jsonl`（含 `thinking_text`、`model=`）。
- **参数**：`query`（必填）、`top_k`、`include_tool_results`（默认 false）。

### 6.3 Agent system 与对话分离

| 内容 | 是否进入 messages.jsonl | 每轮是否进 API |
|------|-------------------------|----------------|
| `build_system_prompt()` | 否 | 是（合并进首条 system） |
| `<session-manifest>` | 是（1 条置顶） | 是 |
| `<conversation-anchor>` | 是（1 条置顶） | 是 |
| 远早 Human/AI/Tool | 是（落盘完整） | 否（出站窗口 + 压缩后删除） |

### 6.4 推荐配置示例（长会话 / 省 Token）

```json
"context": {
  "auto_compress_ratio": 0.55,
  "compress_trigger_max_tokens": 48000,
  "keep_recent_turns": 4,
  "keep_recent_tool_messages": 10,
  "incremental_tool_prune": true,
  "compress_tool_mask_max_chars": 500,
  "dispatch_keep_user_turns": 4,
  "session_history_search_enabled": true,
  "session_history_search_top_k": 8,
  "session_archive_on_compress": true,
  "spill_enabled": true
}
```

用 `/context` 查看当前会话分项 token 估算与压缩阈值。

## 7. 压缩流程（Tier1～3）

### 7.1 触发条件

```text
estimate_tokens(messages) >= resolve_auto_compress_threshold()
```

阈值 = `min(max_tokens × auto_compress_ratio, compress_trigger_max_tokens)`；大窗口模型未配 `compress_trigger_max_tokens` 时默认再 cap 64k。

每轮 `invoke_agent` 在发模型前自动检查；也可交互执行 **`/compress`**（`force=True`）。

### 7.2 流程图

```text
                    ┌─────────────────────────┐
                    │ 当前 messages（全量）   │
                    └───────────┬─────────────┘
                                │
              保留置顶 manifest（若有）
              剥离旧 anchor / 旧 summary
                                │
                                ▼
              ┌─────────────────────────────────────┐
              │ Tier1: split_messages_for_compress     │
              │  · 按 token_budget + keep_recent_turns │
              │    切分为 to_compress | to_keep        │
              │  · 段边界保持 tool 链完整              │
              └───────────┬─────────────────────────┘
                          │
              to_compress 段内超长 Tool → 掩码指针
                          │
              ┌───────────▼─────────────────────────┐
              │ 导出 archive jsonl（可选）            │
              └───────────┬─────────────────────────┘
                          │
              ┌───────────▼─────────────────────────┐
              │ Tier2: run_anchor_update               │
              │  · LLM 摘要 to_compress → 章节增量   │
              │  · 合并进 conversation_anchor.json     │
              │  · 生成 <conversation-anchor>          │
              └───────────┬─────────────────────────┘
                          │
              ┌───────────▼─────────────────────────┐
              │ Tier3（可选）: search_code_hybrid      │
              │  · 对挤出段做代码检索 → related_code   │
              └───────────┬─────────────────────────┘
                          │
              new_messages = [manifest, anchor, *to_keep]
                          │
              update_state + save_session_messages
              sync_session_manifest_after_compress
```

### 7.3 压缩后上下文形态

```text
[ System: <session-manifest> ]
[ System: <conversation-anchor> ]   ← 结构化任务摘要
[ Human / AI / Tool ... ]           ← 仅最近若干轮（to_keep）
```

被挤出的原文在 **`sessions/<thread_id>.jsonl` 归档**；细节通过 `search_session_history` 或 `read_file` 获取。

### 7.4 压缩相关配置

| 配置 | 默认（代码） | 说明 |
|------|-------------|------|
| `auto_compress_ratio` | 0.65 | 相对模型窗口的比例阈值 |
| `keep_recent_turns` | 4 | 至少保留的 user 轮数 |
| `keep_recent_token_ratio` | 0.25 | 保留段 token 预算上限比例 |
| `compress_model` | null | 摘要用的模型；null 用当前模型 |
| `compress_retrieval_enabled` | true | Tier3 代码检索 |
| `compress_retrieval_top_k` | 5 | Tier3 条数 |
| `session_archive_on_compress` | true | 是否写归档 jsonl |

## 9. 运维与排障

| 现象 | 排查 |
|------|------|
| 换模型后 400 / tool 链错误 | 看出站 profile；`scripts/check_model_dispatch_compat.py --probe-api` |
| 压缩后「失忆」 | 读置顶 anchor；`search_session_history`；`read_file` archive |
| 磁盘有 150 条但 API 仍很大 | 看 `dispatch_keep_user_turns` 是否为 0；是否未触发压缩 |

| canonical 迁移 | `scripts/migrate_session_canonical_v2.py` |

## 10. Prompt Cache（`cache_control`）

### 10.1 机制

网关支持 Anthropic Prompt Caching 时，在请求中为**稳定前缀**打 `cache_control` 断点；下一轮请求前缀字节一致则 `cache_read_input_tokens` 上升、计费降低。

llgraph 出站时（`normalize_messages_for_llm`）：

| 断点 | 内容 | 是否缓存 |
|------|------|----------|
| 1 | `build_system_prompt()` 稳定规范 | ✅ `cache_control: ephemeral` |
| 2 | `<session-manifest>` + `<conversation-anchor>` | ❌ 每轮/压缩后可变 |
| 3 | 最后一个 tool 定义（`tag_tools`） | ✅ |
| 4 | 对话中最后一个可缓存块（`llm.bind(cache_control)`） | ✅ 利于最近 N 轮重叠前缀 |

落盘 canonical **不变**；仅发往 API 的 payload 带断点。

### 10.2 配置（`agent.json` → `context.prompt_cache`）

```json
"prompt_cache": {
  "enabled": true,
  "ttl": "5m",
  "tag_tools": true,
  "tag_conversation_tail": true,
  "min_messages": 0,
  "models": ["claude", "gpt"]
}
```

| 字段 | 含义 |
|------|------|
| `enabled` | 总开关 |
| `ttl` | `5m` 或 `1h` |
| `tag_tools` | 缓存工具 schema 块 |
| `tag_conversation_tail` | 绑定 LLM 自动缓存对话尾部可缓存块 |
| `models` | 可选；仅列出的模型 id 子串启用 |

### 10.3 观测

- `/trace token on`：步骤行 `cache 读 …` / `cache 写 …`
- 执行日志 `cache_read_input_tokens`

若长期 `cache 无`，检查网关是否支持 prompt caching，或 manifest 是否每轮大幅改写导致可变块过长（稳定块仍应可命中）。

## 11. 与 Cursor 的对照（摘要）

| Cursor（产品层） | llgraph |
|------------------|---------|
| 滚动摘要 + 短窗口 | anchor + `dispatch_keep_user_turns` + `keep_recent_turns` |
| 代码索引按需拉 | `search_code_hybrid` + spill + read_file |
| 聊天历史不全量重放 | 压缩 + 出站裁剪 + `search_session_history` |
| Prompt cache | `cache_control` 断点（system 稳定块 + tools + 对话尾） |
| 规则/技能目录 | manifest.json + read_file SKILL.md |

更完整对照见 [cursor-agent.md](cursor-agent.md)。
